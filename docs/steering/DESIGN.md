# Mapping Assistant Design Guidelines

> Generated by `steer-design`. Refine this document as the design system evolves вЂ” do not regenerate from scratch.

Last updated: 2026-06-24

## Design Principles

1. **Visually clear** вЂ” The mapping workspace must convey the complete picture at a glance: which fields are mapped, which are not, and which transformation rules apply вЂ” without navigating elsewhere.
2. **Administrator-first simplicity** вЂ” Every UI decision is optimised for the technical administrator's workflow. Avoid unnecessary options, labels, or interface elements that do not support the mapping task.
3. **Everything on one screen** вЂ” All core functionality (source schema, target schema, mapping canvas, AI suggestions) coexists on a single screen. Cross-page navigation is the exception, not the norm.

## Design System

- **Component library:** None вЂ” all UI components are hand-built
- **CSS framework:** Tailwind CSS v4 (configured via `@theme` in CSS, no config file)
- **Connection lines:** Custom SVG bezier curves (`ConnectionLines.vue`) вЂ” not a library
- **Figma library:** Not established вЂ” the orientation sketch in `docs/design/orientation-design.png` is the primary visual reference
- **Storybook:** Not established

## Required UI States

Every interactive feature must support the following states unless explicitly excluded:

- **Default** вЂ” the baseline state
- **Loading** вЂ” async operation in progress (schema fetch, AI suggestion request)
- **Error** вЂ” something went wrong (inline, not in a modal, unless data loss is involved)
- **Empty** вЂ” no fields mapped yet, or schema not yet loaded
- **Disabled** вЂ” action not available in the current context
- **Focus** вЂ” keyboard navigation active (must be visually distinct вЂ” use `focus-visible:ring-2`)

## Accessibility

No automated a11y test suite exists. Current approach is manual `aria-*` usage:

- `aria-hidden="true"` вЂ” SVG overlay (`ConnectionLines.vue`) and decorative icon SVGs
- `aria-label` вЂ” search inputs, icon-only buttons (clear, close, delete)
- `aria-selected` вЂ” `FieldNode.vue` for field selection state
- `aria-pressed` вЂ” `MismatchCard.vue` toggle button
- `role="alert"` вЂ” import error toasts
- `role="status"` вЂ” import warning toasts and validation badges
- `role="group"` вЂ” filter button group in `SourceSchemaPanel.vue`
- `focus-visible:ring-2` вЂ” `ExportButton.vue` and `ImportButton.vue`

Target: WCAG 2.1 AA. Automated a11y testing is not yet in place.

## Orientation Sketch

The orientation sketch (`docs/design/orientation-design.png`) shows the intended UI and is the primary visual reference for the three-panel layout.

![Orientation sketch](../design/orientation-design.png)

What the sketch shows:
- **Left panel** вЂ” source schema (e-Suite) as an expandable field tree; each field shows name, data type (str/num/obj/arr/date/enum) and a REQ badge for required fields
- **Centre panel** вЂ” target schema (OpenZaak) with the same layout; connection lines run from source field to target field
- **Right panel** вЂ” mapping overview with active couplings, status (OK / Warning), transformation rules, and an "Unmapped Required Fields" section
- **Top bar** вЂ” Export, Clear, and Auto-Map actions
