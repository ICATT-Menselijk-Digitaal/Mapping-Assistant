# Mapping Assistant QA Standards

> Generated by `steer-qa`. Refine this document as the test strategy evolves — do not regenerate from scratch.

Last updated: 2026-06-24

## Test Strategy

| Level | Type | Tool | Purpose |
|---|---|---|---|
| Unit | Composables, utility functions, domain logic | Vitest + jsdom | Validate isolated logic without UI |
| Component | Vue components with user interactions | Vitest + Vue Test Utils | Validate rendering and interactions per component |
| E2E | Administrator flows through the application | Playwright | Validate end-to-end behaviour from the administrator's perspective |

## Coverage Requirements

- **New and changed code:** 80% line + branch coverage is the target — *not yet enforced in config; no thresholds are defined in `vitest.config.ts`*
- **Public functions:** at least one happy-path and one error-path test per function
- **Bug fixes:** must include a regression test that would have caught the bug earlier

## Test Patterns

- **Frameworks:** Vitest + Vue Test Utils (unit and component), Playwright (E2E)
- **Test file location:** `__tests__/*.spec.ts` directories colocated with the source folder they cover (e.g. `components/canvas/__tests__/`, `composables/__tests__/`, `utils/__tests__/`)
- **E2E tests:** `frontend/e2e/` — currently only a scaffold placeholder; no real flows covered yet
- **Mock boundary:** mock only at system boundaries (AI API calls, file system). Do not mock internal application code.
- **AI API:** the only external boundary — mock the AI service in unit and component tests

## Gherkin → Test Mapping

- Every Gherkin scenario is covered by at least one automated test
- Test names reference the scenario name so failures are traceable
- Given → test setup, When → action, Then → assertion
- Scenarios describe observable behaviour, not implementation details

## Test Commands

```bash
# Unit + component tests (watch mode)
cd frontend && npm run test:unit

# Unit + component tests (single run, as used in CI)
cd frontend && npm run test:unit -- --run

# E2E tests (requires dev server running)
cd frontend && npm run test:e2e

# Lint
cd frontend && npm run lint

# Type check
cd frontend && npm run type-check
```

## CI Gates (on every PR and push to `main`)

| Job | Command | Blocks merge? |
|---|---|---|
| Lint | Oxlint + ESLint + Prettier check | Yes |
| Type check | `vue-tsc --build` | Yes |
| Unit tests | `npm run test:unit -- --run` | Yes |
| Build | `npm run build-only` | Yes |
| E2E tests | *(disabled — commented out in CI workflow)* | No |

## Definition of Done

- [ ] Linting passes (`npm run lint:ci`)
- [ ] Formatting passes (`npm run format:ci`)
- [ ] Type check passes (`npm run type-check`)
- [ ] All Gherkin scenarios correctly implemented and covered by tests
- [ ] Edge cases handled
- [ ] Contracts match implementation
- [ ] Observability in place
- [ ] Design reviewed (if UI)
- [ ] PR approved by at least one reviewer
- [ ] No unrelated changes or scope creep
- [ ] Documentation updated (if this task changes something described in `docs/steering/` or `docs/glossary.md`)
- [ ] No `test.only` left in the codebase

## Known Issues

- **E2E coverage gap:** The only E2E test (`e2e/vue.spec.ts`) is the Vue scaffold placeholder — it checks for "You did it!" text and does not cover any real administrator flow. E2E is also disabled in CI. Real E2E coverage is not yet in place.
- **No coverage enforcement:** Coverage thresholds are not configured in `vitest.config.ts` — the 80% target is a convention, not a hard gate.

## Roadmap: Incoming Tester

A dedicated QA engineer will be joining the project team. Their primary focus will be establishing real E2E coverage. When they join, the following work is prioritised:

1. **Replace the scaffold placeholder** — delete `e2e/vue.spec.ts` and write real Playwright scenarios based on the Gherkin acceptance criteria in each task
2. **Re-enable E2E in CI** — uncomment the E2E job in `.github/workflows/ci.yml` once at least the core mapping flows are covered
3. **Enforce coverage thresholds** — add a `coverage` block to `vitest.config.ts` once a baseline measurement is established
4. **Exploratory testing** — walk through the real-world use case mapping (the example casing that motivated the project) and document any gaps not caught by automated tests

Playwright is already configured for Chromium, Firefox, and WebKit. The infrastructure is in place — it just needs real test scenarios.
