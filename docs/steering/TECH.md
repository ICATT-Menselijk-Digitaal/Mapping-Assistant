## Architecture

- **Pattern:** Phase 1 — frontend-only SPA (Vue 3 + Vite). Phase 2 — separate frontend SPA and backend REST API.
- **Data flow (Phase 1):** Vue 3 frontend only; schema data and app state are loaded from and persisted to local files via export/import. The frontend calls the Claude API directly for mapping suggestions.
- **Data flow (Phase 2):** Vue 3 frontend → REST calls → .NET Core API → Entity Framework (persistence) + Claude API (AI suggestions).
- **AI integration (Phase 1):** The frontend calls the Claude API (Anthropic) directly for three capabilities: (1) field mapping suggestions with confidence scores via a two-layer approach (embedding-based candidate retrieval followed by LLM reasoning); (2) JSONata transformation expression suggestions for type/format-incompatible field pairs; (3) validation rule inference from OpenAPI field descriptions. Prompt caching keeps repeated calls against the same loaded spec fast. Schema compression (stripping non-essential fields) and an embedding index are built on schema load to handle large specs without exceeding context limits.
- **AI integration (Phase 2):** AI calls move to the backend; the .NET Core API orchestrates them.
- **Layer order:** Domain → Types → Composables + Utils → Components/Views. No services or stores layer currently — business logic lives in composables.

## Key Constraints

- **Components receive `Schema` instances, not `SchemaField[]` arrays.** Pass a `Schema` as a prop; use `Schema` methods (`all`, `byId`, `childrenOf`, `parentOf`, `pathOf`) for all tree traversal. Direct array iteration misses nested fields.
- **Never traverse `SchemaFieldNode` outside `domain/schema.ts`.** It is an internal parsing artefact. The public surface is `Schema`.
- **Code and documentation in English; UI text in Dutch.** All identifiers, comments, type names, and documentation are written in English. User-facing text rendered in the interface (labels, button text, error messages, tooltips) is written in Dutch — the application is used by Dutch municipality personnel.
- **Mock only at system boundaries.** In tests, mock the AI API call; do not mock internal composables or utilities.
- **JSONata is the transformation expression standard.** Do not introduce a second expression language. JSONata expressions are evaluated in-browser for live preview.

## Code Conventions

- **Language split:** Code, identifiers, and documentation → English. UI-facing text rendered to the user → Dutch.
- **Path alias:** `@` maps to `src/` — use `@/composables/...`, `@/types/...` etc.
- **Naming:** camelCase for TypeScript/Vue logic, PascalCase for Vue component names, kebab-case for `.vue` filenames.
- **Test file location:** `__tests__/*.spec.ts` directories colocated with the source folder they cover.

## Domain Model (code-level)

User-facing Dutch domain terms live in [VISION.md](VISION.md#bounded-contexts). The list below is the code-level vocabulary. Add new terms here when introducing a concept.

| Term | Location | Description |
|---|---|---|
| `Schema` | `src/domain/schema.ts` | Parsed representation of one OpenAPI spec. Exposes fields through a queryable interface (`byId`, `has`, `all`, `roots`, `childrenOf`, `parentOf`, `pathOf`, `name`). Read-only after parse. Built via `parseOpenApiSchema(spec)`. |
| `SchemaField` | `src/types/schema.ts` | One field within a schema (`id`, `name`, `path`, `dataType`, `required`, `maxLength?`, `description?`). Hierarchy is not on the field — query it via `Schema` methods. |
| `SchemaFieldNode` | `src/domain/schema.ts` | Internal tree node used during parsing only. Consumers must not traverse it. |
| `DataType` | `src/types/schema.ts` | `'string' \| 'number' \| 'boolean' \| 'date' \| 'object' \| 'array' \| 'unknown'` |
| `FieldMapping` | `src/types/mapping.ts` | One source-to-target field pairing plus its transformations. UI calls this a "koppeling"; code calls it a mapping. |
| `TransformationRule` | `src/types/mapping.ts` | One rule applied as part of a mapping. Kinds: `direct \| static \| expression \| truncate \| default \| cast \| date-format`. |
| Validation status | `src/utils/validationStatus.ts` | `'compatible' \| 'constrained' \| 'incompatible'` for a source/target field pair. |
| AI suggestion | `src/types/ai.ts` + `src/composables/useAISuggestions.ts` | Claude API-backed candidate mapping (source field → target field, confidence score). |
| `MappingSetExported` | `src/domain/events/MappingSetExported.ts` | Domain event fired when a mapping set is exported. |
| `MappingSetImported` | `src/domain/events/MappingSetImported.ts` | Domain event fired when a mapping set is imported. |

### Architectural seams

- **Domain seams live in `src/domain/`.** `Schema` is the primary one. Consumers must not reach past it.
- **Composables (`src/composables/`)** are the Vue-reactive wrappers around the domain. They own loading, error, and lifecycle. Current composables: `useAISuggestions`, `useExport`, `useImport`, `useMappings`, `useSourceSchema`, `useTargetSchema`, `useTransformationSuggestions`.
- **Components (`src/components/`)** receive `Schema` references as props, not raw field arrays. Tree-aware queries go through `Schema` methods.
- **No stores yet.** Pinia is installed for Phase 2 readiness. All reactive state currently lives in composables.

## Architecture Decision Records

| Decision | Rationale | Date |
|---|---|---|
| Vue 3 + .NET Core Web API | Company standard stack; aligns with team expertise and future maintainability | 2026-04-16 |
| Phase 1: frontend-only (no backend) | The PoC phase requires no backend persistence; app state is saved and restored via local file export. AI integration runs directly from the frontend. Backend and EF are added in Phase 2. | 2026-04-16 |
| Claude API for AI suggestions | Strong structured-data reasoning; suitable for field-to-field mapping inference | 2026-04-16 |
| JSONata as transformation standard | AI suggestions generate JSONata expressions. JSONata is readable, evaluable in the browser (live preview), and matches AI output without an extra translation layer. | 2026-04-30 |
| `Schema` as domain seam, not `SchemaField[]` as prop shape | Components receive a `Schema` instance instead of a field array. Reason: top-level array iteration missed nested fields, causing counter and lookup bugs. Tree traversal is now hidden behind `Schema` methods (`all`, `byId`, `childrenOf`, …). | 2026-05-11 |
